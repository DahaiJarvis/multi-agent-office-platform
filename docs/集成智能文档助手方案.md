## 一、项目背景与目标

### 1.1 项目背景

当前多Agent办公平台在对比分析中暴露出以下核心缺陷：

| 优先级 | 缺陷项 | 影响范围 |
|-------|--------|---------|
| P1 | 无RAG知识库 | 无法基于企业私有数据回答问题 |
| P1 | 无文档处理 | 无法解析和索引企业文档 |
| P1 | MCP服务均为模拟 | 所有MCP工具返回模拟数据，无真实业务能力 |
| P1 | Prompt注入防护有限 | 安全防护体系不完善 |
| P2 | 无多模态理解 | 无法处理图片类输入 |
| P2 | 无企业搜索 | 缺少深度检索能力 |

智能文档助手项目已具备完整的RAG知识库、文档处理、深度融合检索、智能路由等能力，且本平台MCP集成层已预留 `knowledge` 服务入口，架构天然契合。

### 1.2 集成目标

1. **RAG知识库构建**：将智能文档助手的ChromaDB向量存储 + BM25 + 知识图谱三路检索能力接入主平台
2. **文档处理能力**：将PDF/Word/Excel/Markdown/TXT/CSV多格式解析 + 语义分块能力接入主平台
3. **深度融合检索**：将向量检索 + BM25关键词 + 知识图谱多跳 + 查询改写 + RRF融合排序能力接入主平台
4. **智能路由增强**：在主平台Supervisor中增加知识/文档相关意图的路由规则

### 1.3 独立性要求

| 维度 | 主平台 | 智能文档助手 |
|------|--------|-------------|
| 代码库 | `/Users/zhaohaipeng/zhp/AI大模型/codes/企业级多Agent办公平台/multi-agent-office-platform` | `/Users/zhaohaipeng/zhp/AI大模型/codes/多文档问答Agent/intelligent-document-assistant` |
| 技术栈 | FastAPI + AutoGen + PostgreSQL | Flask + LangChain + ChromaDB + SQLite/PostgreSQL |
| 运行端口 | 8000 (API) + 9001-9010 (MCP) | 5000 (API) + 9010 (MCP SSE，新增) |
| 数据库 | PostgreSQL (agent_platform) | SQLite/PostgreSQL (intelligent_doc_assistant) |
| 认证体系 | FastAPI-JWT | Flask-JWT |
| LLM供应商 | 通义千问 (DASHSCOPE) | 通义千问 (DASHSCOPE) |
| 部署方式 | Docker Compose | Docker Compose |

两个系统通过MCP协议（SSE传输）通信，不侵入彼此核心代码，各自独立部署、独立运维。

### 2.2 通信机制设计

两个系统间通过两种通信机制交互：

#### 2.2.1 MCP协议通信（Agent工具调用）

```
主平台 Agent ──SSE──> 智能文档助手 MCP Server (9010)
                        │
                        ├── knowledge_search (知识库检索)
                        ├── document_parse (文档解析)
                        ├── document_summary (文档摘要)
                        ├── document_compare (文档对比)
                        ├── web_search (网络搜索)
                        ├── image_analyze (图片分析)
                        ├── report_generator (报告生成)
                        └── kb_manage (知识库管理)
```

**协议细节**：
- 传输方式：SSE (Server-Sent Events)
- 端口：9010
- 端点：`http://<ida-host>:9010/sse`
- 协议版本：MCP 1.0
- 认证方式：API Key（通过MCP请求头传递）

#### 2.2.2 REST API通信（前端直连）

```
主平台前端 ──HTTP──> 智能文档助手 REST API (5000)
                      │
                      ├── /api/knowledge-bases (知识库CRUD)
                      ├── /api/knowledge-bases/<id>/documents (文档上传)
                      ├── /api/qa/parse-files (文件解析)
                      └── /api/qa/analyze-image (图片分析)
```

**协议细节**：
- 传输方式：HTTP/HTTPS
- 端口：5000
- 认证方式：JWT Token（通过Authorization头传递，需跨系统认证映射）

### 2.3 认证体系设计

两个系统JWT实现不同（Flask-JWT vs FastAPI-JWT），采用API Key互信方案：

```
┌──────────────────┐                    ┌──────────────────┐
│    主平台          │                    │  智能文档助手      │
│                  │                    │                  │
│  用户登录 ──JWT──>│                    │                  │
│       │          │   MCP调用           │                  │
│       │          │ ──API Key─────────> │ 验证API Key      │
│       │          │                    │ 执行工具          │
│       │          │ <──JSON Result──── │                  │
│       │          │                    │                  │
│  前端请求         │   REST调用          │                  │
│  JWT Token ──────│──JWT映射──────────>│ 验证映射Token     │
│                  │                    │ 执行API          │
└──────────────────┘                    └──────────────────┘
```

**认证方案**：

| 通信路径 | 认证方式 | 实现细节 |
|---------|---------|---------|
| MCP工具调用 | API Key | 智能文档助手MCP Server启动时加载预共享API Key，主平台MCP Integration Layer在SSE连接时通过请求头传递 |
| REST API调用 | JWT Token映射 | 主平台后端作为代理，将主平台JWT中的用户信息提取后，使用智能文档助手的JWT Secret签发新Token |
| 前端直连 | Nginx反向代理 + Cookie | 通过Nginx统一入口，前端无感知跨系统调用 |

### 3.3 数据安全策略

| 安全维度 | 实施措施 | 说明 |
|---------|---------|------|
| 传输加密 | HTTPS/TLS | 生产环境所有通信走TLS |
| 认证鉴权 | API Key + JWT映射 | 双重认证机制 |
| 数据隔离 | 知识库权限映射 | 主平台用户ID映射到文档助手用户ID |
| 敏感过滤 | 复用文档助手敏感数据过滤器 | 检测PII信息并脱敏 |
| 注入防护 | 复用文档助手Prompt注入检测器 | 检测并拦截恶意输入 |
| 审计日志 | 双系统各自记录 + 关联ID | 通过correlation_id关联两个系统的审计日志 |

### 7.5 回滚机制

| 回滚场景 | 回滚策略 | 操作步骤 |
|---------|---------|---------|
| MCP Server异常 | 禁用knowledge MCP服务 | 1.设置`MCP_SERVER_REGISTRY["knowledge"].enabled = False` 2.重启agent-api |
| 文档助手服务宕机 | 降级为原有模拟服务 | 1.停止ida-backend 2.恢复knowledge MCP Server为原有模拟实现 |
| 集成后性能下降 | 回滚到集成前版本 | 1.git revert集成提交 2.重新部署主平台 3.文档助手独立运行不受影响 |
| 数据异常 | 恢复数据库快照 | 1.停止服务 2.恢复PostgreSQL快照 3.恢复ChromaDB数据目录 |

**关键回滚原则**：
- 两个系统独立部署，回滚任一系统不影响另一个
- 主平台回滚时，文档助手继续独立运行
- 文档助手回滚时，主平台降级为模拟knowledge服务

## 九、风险评估

### 9.1 技术风险

| 风险项 | 概率 | 影响 | 应对措施 |
|-------|------|------|---------|
| MCP SSE连接不稳定 | 中 | 高 | 实现自动重连机制，设置连接超时和重试策略；主平台MCP Integration Layer已有缓存和错误处理 |
| JWT跨系统认证复杂 | 中 | 中 | 优先采用API Key方案（MCP调用），REST代理由主平台后端签发新Token，避免直接Token互认 |
| 文档助手服务宕机影响主平台 | 低 | 高 | 主平台MCP Integration Layer已有服务禁用机制，knowledge服务不可用时自动降级 |
| ChromaDB并发性能瓶颈 | 中 | 中 | 文档助手已实现LRU缓存和连接池，必要时升级为Milvus |
| 两个系统LLM调用冲突 | 低 | 低 | 两系统均使用DASHSCOPE，模型供应商一致；各自独立调用，无共享状态 |
| 数据库连接池耗尽 | 低 | 中 | 两个系统独立数据库，连接池互不影响；文档助手已配置pool_size和max_overflow |

### 9.2 业务风险

| 风险项 | 概率 | 影响 | 应对措施 |
|-------|------|------|---------|
| 用户权限映射错误 | 中 | 高 | 实现严格的权限校验，知识库操作前二次验证用户身份；文档助手已有viewer/editor/admin权限体系 |
| 知识库数据泄露 | 低 | 高 | 文档助手已有公司/部门/个人三级权限；集成时复用其权限服务；传输全程TLS |
| 集成后响应延迟增加 | 中 | 中 | MCP SSE本地调用延迟<10ms；同机房部署减少网络延迟；实现检索结果缓存 |
| 两个系统数据不一致 | 低 | 低 | 两个系统数据独立，不共享数据库；用户操作各自记录审计日志 |

### 9.3 运维风险

| 风险项 | 概率 | 影响 | 应对措施 |
|-------|------|------|---------|
| 部署复杂度增加 | 高 | 中 | 统一Docker Compose管理；编写详细部署文档；实现一键启动脚本 |
| 监控盲区 | 中 | 中 | 文档助手已有/monitoring/health端点；集成到主平台Grafana监控面板 |
| 日志分散 | 中 | 低 | 两个系统各自输出日志；通过correlation_id关联；建议统一日志收集到ELK |
| 版本升级不同步 | 中 | 中 | MCP协议接口稳定，版本升级不影响；制定联合升级计划 |

### 9.4 风险缓解优先级

```
高优先级（必须解决）：
1. MCP SSE连接稳定性 → 实现重连机制 + 降级策略
2. 用户权限映射 → 严格校验 + 二次验证
3. 服务宕机影响 → 自动降级 + 健康检查

中优先级（需要关注）：
4. JWT跨系统认证 → API Key优先 + 代理签发
5. 响应延迟 → 缓存 + 同机房部署
6. 部署复杂度 → Docker Compose统一管理

低优先级（持续优化）：
7. 日志关联 → correlation_id
8. 监控集成 → Grafana面板
9. 版本同步 → 联合升级计划
```
